1 · Lifecycle of one review

A PR (or local diff) becomes a run directory, gets annotated by the augment pipeline, is served to a browser by an ephemeral HTTP server, and — for scr pr — posts the reviewer's comments back to GitHub as one review.

4 · Augment pipeline

Three passes run in sequence: overview (one call per PR), hunk (one call per hunk, concurrent), extra-review (one optional call per PR). A fourth pass — fold-summary — fires on demand from the review server when a reviewer collapses a region. All four go through one deep Pass module that owns the LLM-call recipe.

The hunk pass is the only one that takes deps — a RepoTools handle exposing read_file / grep / list_dir / git_log over the run's worktrees. SDK backends get it via pydantic-ai's deps; CLI drivers expose the same tools over MCP.

5 · Backend / Client / Driver

The CLI resolves a backend name to a Backend adapter; the adapter resolves credentials and produces a Client that the augment pipeline drives. Two implementation families sit behind one Backend interface: SDK backends wrap a pydantic-ai Model; CLI drivers spawn a third-party CLI subprocess on every request().

two families behind Backend

SDK family
anthropic-sdk · gemini-api · openai-compat

Wraps a pydantic-ai Model (AnthropicModel, GoogleModel, …). Credentials via env var or api_key_command. Tool-use loop runs in-process.

CLI driver family
claude-cli

The driver is a custom pydantic_ai.Model subclass — extends SubprocessModel base. Spawns claude -p per request. Tool-use loop runs inside the subprocess via MCP, not in pydantic-ai.

The Backend seam is real — five adapters live there today. register_handler() is the test-time substitution hook; production code never calls it.

6 · Format & schema

AnnotatedDiff is the central schema. Three siblings in format/ serialise it: parse reads a unified diff (with or without annotations), emit writes the canonical text form, sidecar round-trips it as JSON. The contract is parse(emit(x)) == x.

The TS viewer consumes ViewerData as /data.json. The persisted shapes (augmented.diff + sidecar) are the wire format and the structural shape respectively — kept in sync by any code that mutates one (e.g. the on-demand fold-summary pass).

7 · Review server

Ephemeral localhost ThreadingHTTPServer spawned by scr review / scr pr. Dies on POST /exit from the viewer or after an idle timeout. No external HTTP deps.

routes

method path does what
GET/viewer html
GET/data.jsonViewerData snapshot
GET/static/*viewer assets (asset whitelist)
GET/eventsSSE stream — buffered + Last-Event-ID replay
POST/fold-summaryfires the fold-summary pass; patches viewer json
POST/commentadd reviewer comment → comments.json
DELETE/commentremove reviewer comment
POST/exitgraceful shutdown

Each connected viewer gets a blocking queue. Publish never blocks the augment pipeline.

8 · Seams worth naming

Five places where behaviour swaps without editing in place. Each one earns its keep — two or more adapters in production + tests.

Backend seam
5 adapters

SDK + CLI families. Tests substitute via register_handler().

RunSpec seam
2 sources

fetch/github and fetch/local both produce a RunSpec; shared materialize_run_metadata writes the dir.

Pass recipe seam
4 passes

augment/pass_.py · overview · hunk · fold-summary · extra-review.

AnnotatedDiff seam
3 serialisers

format/parse, format/emit, format/sidecar all bridge to one schema.

RepoTools seam
2 transports

augment/tools.py defines the read-only repo surface (read_file, grep, list_dir, git_log) once. SDK backends receive it via pydantic-ai's deps; CLI drivers expose the same tools over MCP. One source of truth, two wire formats.
